# Data Dictionary and Provenance

This document describes every dataset shipped with the repo, its origin
(real vs. synthetic / derived), and the role each one plays in the pipeline.

> **Reproducibility contract:** every artifact under `data/processed/`,
> `data/interim/`, `outputs/metrics/`, `outputs/figures/` is *regeneratable*
> from `data/raw/` via the scripts in `scripts/`. Random seed is fixed at
> `RANDOM_SEED = 42` (see `src/phm/config.py`).

---

> **Provenance + target correction (lab-confirmed 2026-06-13, P7.3):** the dataset is
> **in-house laboratory data**, NOT the NUAA/PHM 2010 challenge. The official modeling target is
> **VB** (50 kHz sampling, confirmed). The official VB source is
> **`data/targets/microscope_vb.csv`** (column `VB_um`, 12 experiments) — this is the file
> formerly named `microscope_vs.csv` whose `VS` column was a misnamed VB column, now renamed.
> `vb_targets.csv` is a **legacy/model-ready subset** (10 experiments with recorded signals)
> used by the P1–P6 baseline. See `config/targets.yaml` and `reports/p7_3_gate_closure_summary.md`.

### Why some data is synthetic

The augmented rows under `data/interim/augmentation/` are **only used during
training inside cross-validation folds** to mitigate the n = 10 sample size
problem. Test rows are *always* the real microscope-measured VB values.
This is explicitly documented in `src/phm/layered_pipeline.py` (`run_branch`)
and verified by the LOEO-CV split logic in `src/phm/splitting.py`.

## 2. Modeling dataset — `data/processed/experiment_features.csv`

| Stat                  | Value                                  |
|-----------------------|----------------------------------------|
| Rows                  | 10 (one per experiment)                |
| Columns               | 208                                    |
| Tool                  | T01 (single tool, no replicates)       |
| Target                | `VB_um` (flank wear, micrometers)      |
| Granularity           | One row per experiment                 |
| Numerical columns     | 207                                    |
| Categorical columns   | 1 (`F` — cutting force regime label)   |
| Missing values        | < 1% (imputed with median at fit time) |

### 2.1 Column groups

| Prefix / column     | Count | Meaning                                                                     |
|---------------------|------:|-----------------------------------------------------------------------------|
| `experiment_id`     | 1     | Integer ID (used as LOEO grouping key)                                      |
| `tool_id`           | 1     | Always `T01` in this dataset                                                |
| `experiment_order`  | 1     | Chronological position (1 .. 10)                                            |
| `A_*`               | ~101  | Axial-axis vibration statistics per contact window (p1..p6 = pass 1..6)     |
| `R_*`               | ~99   | Rotational-axis vibration statistics per contact window                     |
| `F`                 | 1     | Force regime label (categorical)                                            |
| `end_of_life`       | 1     | Binary flag for end-of-life experiments                                     |
| `VB_um`             | 1     | **Target** — measured flank wear in micrometers                             |

### 2.2 Feature suffix grammar (axial / rotational signals)

For each contact window `p{n}` (n = 1..6) and each signal channel, we compute:

| Suffix          | Meaning                                       |
|-----------------|-----------------------------------------------|
| `_mean`         | Arithmetic mean of the windowed signal        |
| `_std`          | Standard deviation                            |
| `_rms`          | Root mean square                              |
| `_max`, `_min`  | Extremes                                      |
| `_p2p`          | Peak-to-peak amplitude                        |
| `_skew`, `_kurt`| Higher-order moments                          |
| `_crest_factor` | RMS / peak ratio                              |
| `_energy`       | Sum of squares                                |

## 3. Target — `VB_um`

| Property        | Value                          |
|-----------------|--------------------------------|
| Source          | Optical microscope readings    |
| Unit            | Micrometers (µm)               |
| Range observed  | ~80 - ~280                     |
| Distribution    | Right-skewed, one mode         |
| Replicates      | **None** — one measurement per experiment |

Because every experiment yields exactly one `VB_um`, **leave-one-experiment-out
cross-validation (LOEO-CV)** is the only honest validation scheme. K-fold CV
on a flattened row-level dataset would leak between folds.

---

## 4. Splitting policy

- **LOEO-CV** (Leave-One-Experiment-Out) — `experiment_id` is the grouping key.
- 10 folds, 9 training rows, 1 test row per fold.
- Implementation: `sklearn.model_selection.LeaveOneGroupOut` in
  `src/phm/splitting.py`.
- No hold-out test set — n = 10 is too small to afford one; the LOEO loop
  *is* the test.

---

## 5. Augmentation rules (training rows only)

| Strategy            | Applied to             | What it does                                              |
|---------------------|------------------------|-----------------------------------------------------------|
| `feature_noise`     | Training fold only     | Adds Gaussian N(0, sigma) per feature                     |
| `feature_scaling`   | Training fold only     | Multiplies each feature by U(1-eps, 1+eps)                |
| `grouped_scaling`   | Training fold only     | One scalar per feature group (A_p1, A_p2, ...)            |

Augmented rows **never enter the test set** of any fold. This is asserted in
`src/phm/layered_pipeline.py::run_branch`.

---

## 6. Sensitive data note

The raw `data/raw/` and `data/targets/` folders are git-ignored. The
processed CSVs (`data/processed/experiment_features.csv`,
`data/processed/modeling_dataset.csv`) are regeneratable from raw and are
also git-ignored. Only metric outputs (`outputs/metrics/`) and report
artifacts (`outputs/reports/*.pptx`) are tracked.
